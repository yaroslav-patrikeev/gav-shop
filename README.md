# 🐾 Практическое задание: Комплексное покрытие тестами интернет-магазина «Гав-Шоп»

Добро пожаловать на практическую работу! Твоя задача — спроектировать и написать полноценную пирамиду тестирования для веб-приложения «Гав-Шоп» (магазин игрушек для собак).

Приложение спроектировано по методологии **Feature-Driven Design (FSD)**, разделено на слои и управляет состоянием на клиенте на основе данных, получаемых из внешнего API.

---

## 🎯 Цель задания

Реализовать комплексную стратегию тестирования, защитив ключевые пользовательские сценарии (User Stories) от регрессии на всех уровнях.

---

## 🛠️ Выбор технологий

Ты можешь использовать **любые известные тебе тестовые фреймворки, библиотеки и инструменты** для автоматизации (как для запуска в Node.js, так и для браузерной автоматизации). Главное — обеспечить выполнение технических требований ниже.

---

## 📋 Техническое задание по уровням пирамиды

### Уровень 1: Изолированные тесты логики (Unit)

**Объект проверки:** Модуль управления состоянием корзины.

- **Сценарий 1:** Проверить, что при добавлении нового товара в корзину он успешно появляется в списке, а его начальное количество равно 1.
- **Сценарий 2:** Проверить, что при повторном добавлении _одного и того же_ товара количество увеличивается, но объект в списке не дублируется.
- **Сценарий 3:** Убедиться, что базовая цена за единицу товара в состоянии остается неизменной (не умножается на количество внутри логики обновления данных).

### Уровень 2: Интеграционные тесты взаимодействия (UI + State + Сеть)

**Объект проверки:** Взаимодействие страницы каталога, виджета шапки сайта и сетевого слоя.

- **Требование к изоляции:** Исходный код приложения не должен содержать фейковых заглушек (запросов к локальным массивам). Сетевой слой должен выполнять честные запросы. Имитация ответов сервера должна происходить на уровне тестового окружения.
- **Пользовательский флоу:** Тест эмулирует открытие страницы, дожидается получения данных от сетевого симулятора, совершает клик по кнопке покупки товара и проверяет, что счетчик в шапке сайта мгновенно синхронизировался с новым состоянием приложения.

### Уровень 3: Визуальное тестирование компонентов (Snapshot/Visual Regression)

**Объект проверки:** Корректность отображения и верстки страницы каталога.

- **Требование к изоляции:** Вся инфраструктура для визуальных тестов (включая конфигурационные файлы сборщиков и HTML-шаблоны для рендеринга компонентов) должна лежать строго внутри подпапки визуальных тестов.
- **Сценарий 1:** Снять эталонный снимок страницы со всеми загруженными карточками товаров. Проверить адаптивность и пропорции изображений при рендеринге.
- **Сценарий 2:** Симулировать переключение табов фильтрации товаров (например, по категориям размеров), дождаться изменения интерфейса и зафиксировать снимок обновленного состояния (проверка изменения стилей активных элементов).

### Уровень 4: Сквозные тесты (End-to-End)

**Объект проверки:** Полный цикл покупки в реальном окружении.

- **Пользовательский флоу:** Автоматизировать сквозной путь пользователя в браузере. Зайти на работающий локально сайт -> Применить фильтр -> Найти целевой товар -> Добавить его в корзину -> Убедиться, что сквозной процесс прошел успешно и интерфейс отобразил результат для покупателя.

---

## 🛑 Критерии приемки (Definition of Done)

1. **Полная автономность:** При удалении папки `tests/` и связанных с тестами команд из `package.json` проект компилируется без единой ошибки или варнинга. Продакшен-код в `src/` ничего не знает о тестовой среде.
2. **Стабильность (No Flaky Tests):** В коде тестов запрещено использование явных захардкоженных задержек (вроде `sleep` или `setTimeout` на случайное количество секунд). Ожидание элементов и ответов сети должно быть построено на механизмах предикатов, событий или автоматических умных ожиданий (auto-waiting).
3. **Воспроизводимость:** Все тесты запускаются локально через стандартные CLI-скрипты и проходят без ошибок (`green`).
4. **Конфигурация:** Локальный конфигурационный файл для скриншотных тестов настроен так, чтобы генерировать и сравнивать снимки в рамках своей изолированной директории.
